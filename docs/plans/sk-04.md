# SK-04 Implementation Plan: Content References (Relations)

Spec: `docs/specs/sk-04-design.md`

| File | Changes |
|------|---------|

## Commits

### Commit 1: Make `build_reference_options` recursive

- [ ] **`src/routes/content.rs`**: Refactor `build_reference_options` (lines 282-340):
  - When a property has `type: "object"`, recurse into its `properties` with `prefix.key` path
  - When a property has `type: "array"` with `items` containing reference fields, recurse into `items.properties` using the base field name (no index) as the key in `ReferenceOptions`
  - Add depth parameter, cap at 32 (matching `MAX_NESTING_DEPTH`)
- [ ] **`src/content/form.rs`** (line 389): Update `ref_options.get(name)` lookup to also try stripping ALL array indices -- if `name` is `sections[0].items[1].author`, strip to `sections.items.author` as fallback key. Use a helper that removes all `[N]` patterns: `name.replace(|c: char| c == '[', "").replace(|c: char| c.is_ascii_digit(), "").replace(']', "")` or a regex. Also handle the simple case: `tags[0]` -> `tags`.
- [ ] **`src/content/form.rs`** (tests): Unit test: reference field inside array items gets options populated via stripped-index lookup

### Commit 2: Configurable `label_field`

- [ ] **`src/routes/content.rs`**: In `build_reference_options`, after reading `x-substrukt-reference.schema`, also read `x-substrukt-reference.label_field`
  - If `label_field` is specified: use `entry.value().get(label_field)` for the label string
  - If `label_field` is missing or the field doesn't exist on the entry: fall back to current behavior (first non-`_` string field)
  - Log `tracing::warn` if `label_field` is specified but not found on any entry
- [ ] **Test**: Integration test with schema declaring `label_field: "display_name"`, verify dropdown shows that field value

### Commit 3: Reference search route

- [ ] **`src/routes/content.rs`**: Add route handler `async fn ref_search(...)`:
  - Path: `GET /apps/{app}/content/{schema_slug}/_ref/search`
  - Query params: `target` (target schema slug), `q` (search query), `limit` (default 10)
  - Loads target schema entries from cache (`{app_slug}/{target_slug}/` prefix)
  - If `q` is empty: return most recent entries (sorted by cache insertion order, limited)
  - If `q` is non-empty: filter using `content::matches_query`, limit results
  - Extract label per entry using `label_field` from the referring schema's `x-substrukt-reference` config (passed as query param `label_field`) or first-string-field fallback
  - Return HTML partial with `_ref_search_results.html`
- [ ] **`src/routes/content.rs`**: Register route `.route("/{schema_slug}/_ref/search", get(ref_search))` -- `_ref` is a literal segment, so Axum resolves it before `{entry_id}` wildcard regardless of order. Place near other schema-scoped routes for readability.
- [ ] **`templates/content/_ref_search_results.html`**: New partial template:
  ```html
  {% for entry in results %}
  <div class="ref-option px-3 py-2 hover:bg-card-alt cursor-pointer" data-id="{{ entry.id }}" data-label="{{ entry.label }}">
    <span class="font-medium text-sm">{{ entry.label }}</span>
    <span class="text-muted text-xs ml-2">{{ entry.id }}</span>
  </div>
  {% endfor %}
  {% if results is empty %}
  <div class="px-3 py-2 text-muted text-sm">No entries available.</div>
  {% endif %}
  ```

### Commit 4: Typeahead form component

- [ ] **`src/content/form.rs`**: Add constant `const TYPEAHEAD_THRESHOLD: usize = 50;`
- [ ] **`src/content/form.rs`**: In `render_field` for `("string", Some("reference"))` (line 386-410):
  - Check `ref_options.get(name).map(|o| o.len()).unwrap_or(0)`
  - If <= `TYPEAHEAD_THRESHOLD`: render existing `<select>` (no change)
  - If > `TYPEAHEAD_THRESHOLD`: render typeahead component instead:
    - Hidden input `<input type="hidden" name="{name}" value="{current_id}">` for the actual value
    - Text input with `hx-get` pointing to `/_ref/search?target={target_slug}&label_field={label_field}` with `hx-trigger="input changed delay:300ms"`, `hx-target="#ref-results-{name}"`
    - Results container `<div id="ref-results-{name}">` below the input
    - If current value exists: show as selected chip with clear button
  - Extract `target_slug` and `label_field` from the schema property's `x-substrukt-reference`
- [ ] **`templates/content/_ref_typeahead.html`**: Reusable partial (or inline in form.rs HTML generation):
  - Container with relative positioning for dropdown
  - JavaScript (inline or in base template script block): click handler on `.ref-option` sets hidden input value, shows selected label, hides dropdown; clear button resets
- [ ] **Test**: Manual verification with a schema that has > 50 target entries -- verify typeahead renders instead of select

  4. If valid: proceed

### Commit 8: Delete warning for referencing entries

- [ ] **`src/content/mod.rs`**: Add `find_referencing_entries(cache, schemas_dir, app_slug, target_schema_slug, target_entry_id) -> Vec<(String, String)>`:
  - List all schemas in `schemas_dir`
  - For each schema: check if any property has `format: "reference"` with `x-substrukt-reference.schema == target_schema_slug`
  - If so: scan cache entries for that schema, check if any contain `target_entry_id` as a reference value
  - Return `(schema_slug, entry_id)` pairs of referencing entries
- [ ] **`src/routes/content.rs`**: In `delete_entry` handler (line 809+):
  - Before deleting: call `find_referencing_entries`
  - If referencing entries found: include warning in flash message ("This entry was referenced by N entries in {schema}")
  - Proceed with delete (warn, don't block)
- [ ] **`src/routes/api.rs`**: In `delete_entry` handler (line 531+):
  - Before deleting: call `find_referencing_entries`
  - Include `"warnings"` field in 204->200 response if references found: `{ "status": "deleted", "warnings": ["Referenced by posts/hello-world"] }`
  - Note: changes DELETE response from 204 (no body) to 200 (with JSON body) when warnings exist. This is a minor API behavior change -- consumers checking for 204 specifically may need updating. Consider using a response header (e.g., `X-Substrukt-Warnings`) instead to avoid changing the status code. If using 200, ensure the response is `200 { "status": "deleted", "warnings": [...] }` and document the change.
- [ ] **Test**: Integration test: create author, create post referencing author, delete author via API, verify warning in response
