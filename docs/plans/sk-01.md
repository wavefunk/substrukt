# SK-01: Content Versioning and History — Implementation Plan

Spec: `docs/specs/sk-01-design.md`

## Summary

Most versioning infrastructure exists. This plan implements the remaining gaps: version diffing (field-level comparison), version preview, REST API for history, and authorship metadata via sidecar files.

### Phase 1: Version Diff (core logic)

- [ ] **1.1 Add diff types to `src/history.rs`**

  Add after the existing `VersionInfo` struct:

  ```rust

      pub path: String,
      pub label: String,
      pub kind: DiffKind,
  }

  #[serde(tag = "type", rename_all = "snake_case")]
  pub enum DiffKind {
      Changed { old: serde_json::Value, new: serde_json::Value },
      Added { value: serde_json::Value },
      Removed { value: serde_json::Value },
  }
  ```

  No commit yet — combined with 1.2.

- [ ] **1.2 Implement `diff_entries()` in `src/history.rs`**

  ```rust
  pub fn diff_entries(old: &Value, new: &Value, schema: &Value) -> Vec<FieldDiff>
  ```

  Algorithm:
  1. Walk `schema["properties"]` recursively.
  2. For each property: compare `old[key]` vs `new[key]`. Emit `Changed`, `Added`, or `Removed`.
  3. Recurse into `type: "object"` with dotted paths.
  4. For `type: "array"`, compare element-by-element positionally.
  5. Skip `_status` and `_id` keys.
  6. After schema walk, scan both data objects for keys not in schema (schema drift). Emit as `Added`/`Removed` with key name as label.
  7. Depth limit: 32 (match `MAX_NESTING_DEPTH` from `content/mod.rs:295`).

  **Tests** (add to `src/history.rs` `#[cfg(test)]` block):
  - Identical entries -> empty vec.
  - Changed string field -> one `Changed` entry.
  - Added field (in new, not old) -> `Added`.
  - Removed field (in old, not new) -> `Removed`.
  - Nested object diff -> dotted path.
  - Array element changed -> correct index path.
  - `_status` and `_id` excluded.
  - Schema drift: key in data but not schema.
  - Depth limit: 33-deep nested object -> stops at 32.

  **Commit**: `feat: add diff_entries for field-level version comparison`

- [ ] **2.2 Add diff route to `src/routes/content.rs`**

  Register in `routes()`:
  ```rust

  ```

  Handler `entry_diff`:
  - Parse query params `from` (required u64 timestamp) and `to` (optional u64, defaults to current).

  - Call `history::diff_entries(from_data, to_data, &schema.schema)`.

### Phase 4: API Endpoints

- [ ] **4.1 Add history API handlers to `src/routes/api.rs`**

  Three new async functions:

  `api_list_versions`: Load `history::list_versions()`, format timestamps as ISO 8601 strings via chrono, return JSON array. Auth: viewer+.

  `api_get_version`: Load `history::get_version()`, return raw JSON. 404 if not found. Auth: viewer+.

  `api_revert_version`: Load version, snapshot current, save via `content::save_entry()`, reload cache, audit log. Auth: editor+.

  **Commit**: `feat: add REST API endpoints for version history`

- [ ] **4.2 Register API routes in `api_app_routes()`**

  Add before the existing `/content/{schema_slug}/{entry_id}` route:
  ```rust
  .route(
      "/content/{schema_slug}/{entry_id}/versions",
      get(api_list_versions),
  )
  .route(
      "/content/{schema_slug}/{entry_id}/versions/{timestamp}",
      get(api_get_version),
  )
  .route(
      "/content/{schema_slug}/{entry_id}/versions/{timestamp}/revert",
      post(api_revert_version),
  )
  ```

  These routes won't conflict with the existing `/{entry_id}` catch-all because `versions` is an additional static path segment that Axum matches by specificity. However, they must be registered before the `/{entry_id}` route for clarity and to match the existing pattern (e.g., `/publish` and `/unpublish` are registered before `/{entry_id}`).

  **Tests** (in `tests/integration.rs`):
  - `GET /versions` with no history -> `[]`.
  - Create entry, update it, `GET /versions` -> 1 version.
  - `GET /versions/{ts}` -> returns historical data.
  - `GET /versions/99999` -> 404.
  - `POST /versions/{ts}/revert` -> entry data restored, new snapshot created.
  - Viewer can list/get versions, cannot revert.
  - Editor can revert.

  **Commit**: `feat: register version history API routes with tests`

## Risk Notes

- **API endpoint paths**: `/content/{schema_slug}/{entry_id}/versions` adds a new path segment. Axum routes by specificity, so this won't conflict with the existing `/{entry_id}` catch-all. Verified by existing `/publish` and `/unpublish` routes using the same pattern.
- **Snapshot metadata signature change**: commit 5.2 changes the `snapshot_entry` signature AND updates all existing callers to pass `None`. Commit 5.5 upgrades those `None` values to `Some(meta)`. This two-step approach keeps every commit compilable.
- **Template changes to history.html**: tasks 2.3, 3.3, and 5.6 all modify the same template. These are in sequence, so no merge conflict risk.
